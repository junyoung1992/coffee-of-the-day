# Phase 2-4: Happy-path E2E 자동화

## 무엇을 만들었나

Phase 2-4에서는 사용자가 실제 브라우저에서 밟는 핵심 흐름을 자동으로 검증하는 최소 E2E 테스트를 추가했습니다.

대상 시나리오는 다음 한 줄로 요약할 수 있습니다.

`생성 → 목록 반영 → 상세 → 수정 → 삭제`

이번 자동화는 브루 로그를 기준으로 작성했습니다. 이유는 Phase 2에서 강화된 브루 폼, 레시피 입력, 목록 필터까지 한 번에 검증할 수 있기 때문입니다.

---

## 왜 단위 테스트만으로는 부족했나

기존 프론트엔드는 `vitest` 기반 단위 테스트가 이미 있었습니다. 이 테스트들은 각각의 함수, 훅, 컴포넌트가 "부분적으로" 올바른지 확인하는 데는 충분합니다.

하지만 실제 사용자 흐름은 다음처럼 여러 계층이 연결됩니다.

1. React Router가 페이지 전환을 처리한다.
2. 폼 상태가 API payload로 직렬화된다.
3. 브라우저 `fetch`가 실제 백엔드로 요청을 보낸다.
4. Go 서버가 SQLite에 기록을 저장한다.
5. 다시 목록/상세 API를 읽어 화면에 반영한다.

Java/Spring으로 비유하면, Controller test와 Service unit test만으로는 "브라우저에서 버튼을 눌렀을 때 DB까지 실제로 연결되는가"를 보장하지 못하는 것과 같습니다. 이번 E2E는 그 마지막 연결 구간을 메웁니다.

---

## Playwright를 선택한 이유

`review/refactoring_phase_1.md`에서는 Playwright 또는 Cypress를 후보로 언급했습니다. 이번 구현에서는 Playwright를 선택했습니다.

이유는 세 가지입니다.

1. 설정이 단순합니다. 프론트엔드 프로젝트 안에 바로 붙이기 쉽습니다.
2. `webServer` 설정으로 백엔드와 프론트엔드 dev server를 함께 올릴 수 있습니다.
3. 브라우저 상호작용 API가 안정적이라 happy-path 한 줄기를 빠르게 자동화하기 좋습니다.

즉, 지금 단계에서는 "가장 적은 비용으로 실제 사용자 흐름을 검증하는 도구"라는 점이 중요했습니다.

---

## 서버를 테스트가 직접 띄우도록 만든 이유

`frontend/playwright.config.ts`는 테스트 실행 시 두 서버를 함께 기동합니다.

- 백엔드: `go run ./cmd/server`
- 프론트엔드: `npm run dev -- --host 127.0.0.1 --port 4173`

그리고 프론트엔드는 `VITE_API_BASE_URL`로 E2E 전용 백엔드를 바라보도록 고정했습니다.

구현 중 한 가지 실제 이슈가 있었습니다. 프론트는 `127.0.0.1:4173`에서 뜨는데, 백엔드 CORS는 처음에 `http://localhost:5173`만 허용하고 있었습니다. 이 상태에서는 브라우저 기준으로 cross-origin 요청이 막혀 `Failed to fetch`가 발생합니다.

그래서 최종 구현에서는 백엔드 CORS 허용 목록에 다음 둘 다 포함했습니다.

- `http://localhost:5173`
- `http://127.0.0.1:4173`

즉, 이번 E2E는 단순히 테스트만 추가한 것이 아니라, 로컬 개발 환경에서 주소 표기 차이로 발생할 수 있는 실제 통합 문제도 함께 정리한 셈입니다.

이 방식은 Java/Spring에서 `@SpringBootTest(webEnvironment = RANDOM_PORT)`로 실제 애플리케이션을 띄운 뒤, 외부 클라이언트가 붙는 구조와 비슷합니다. 테스트가 mock server를 붙이는 것이 아니라, 실제 실행 구성을 최대한 유지한 채 검증하는 방식입니다.

## 왜 `POC_SEED_USER_ID`를 추가했나

이 프로젝트는 아직 인증이 없고, `X-User-Id` 헤더를 그대로 신뢰하는 POC 단계입니다. 문제는 `coffee_logs.user_id`가 `users.id`를 참조하므로, 테스트가 로그를 생성하려면 먼저 `users` 테이블에 사용자가 있어야 한다는 점입니다.

그래서 백엔드 시작 시 선택적으로 테스트용 사용자를 보장하는 `POC_SEED_USER_ID` 환경변수를 추가했습니다.

핵심 포인트는 다음과 같습니다.

- 기본값은 비어 있으므로 일반 실행에는 영향이 없습니다.
- 값이 있을 때만 `users` 테이블에 `INSERT OR IGNORE`로 최소 사용자 레코드를 넣습니다.
- E2E에서는 프론트 기본 사용자 ID인 `dev-user`를 그대로 사용할 수 있습니다.

이것은 Spring Boot의 `data.sql`이나 테스트 프로파일용 seed data를 아주 작게 적용한 것과 비슷한 접근입니다.

## 시나리오 설계

이번 테스트는 "브루 로그 happy-path" 하나만 추가했습니다.

검증 내용은 다음과 같습니다.

1. 홈에서 기록 추가 페이지로 이동
2. 브루 폼 입력 후 저장
3. 상세 화면에서 생성 결과 확인
4. 목록으로 돌아가 브루 필터 적용 후 카드 노출 확인
5. 수정 화면에서 주요 필드 변경 후 저장
6. 상세 화면에서 수정 결과 확인
7. 삭제 후 홈으로 복귀하고, 수정된 로그 카드가 더 이상 존재하지 않음을 확인

중요한 점은, 이 한 테스트가 다음을 동시에 검증한다는 것입니다.

- 라우팅
- 폼 직렬화
- 실제 HTTP 요청
- SQLite 저장
- React Query 재조회
- 목록 필터링

즉, 여러 단위 테스트가 나눠서 보던 연결 지점을 실제 브라우저 기준으로 한 번 묶어 확인합니다.

## DB 초기화 전략

Playwright 설정에서는 실행마다 고유한 임시 SQLite 파일 경로를 사용합니다.

경로는 저장소 내부가 아니라 OS 임시 디렉토리 아래에 생성합니다. 또한 `reuseExistingServer: false`로 두어, 테스트를 다시 실행할 때 이전 서버 프로세스와 DB 상태를 재사용하지 않도록 했습니다.

이렇게 한 이유는 테스트 간 상태 오염을 막기 위해서입니다. 인메모리 DB를 직접 주입하는 대신 파일 DB를 쓰는 이유는, 백엔드 서버가 별도 프로세스로 떠 있기 때문입니다. 같은 프로세스 안의 테스트 더블을 주입하는 방식보다 실제 실행 환경과 더 가깝습니다.

Java 테스트로 치면 각 테스트마다 독립적인 test database를 다시 준비하는 셈입니다.

## 테스트 러너 분리

구현 후 정리 과정에서 `vitest`가 `frontend/e2e`와 `node_modules` 내부 테스트 파일까지 함께 읽으려는 문제가 드러났습니다. 이 상태에서는 Playwright 스펙이 단위 테스트 러너에 의해 잘못 실행되거나, 의존성 패키지 내부 테스트까지 잡혀서 테스트가 깨집니다.

그래서 최종적으로 `frontend/vite.config.ts`의 `test.include`를 아래처럼 제한했습니다.

```ts
include: ['src/**/*.{test,spec}.{ts,tsx}']
```

이 설정의 의미는 단순합니다.

- Vitest는 `src/` 아래 단위 테스트만 본다.
- Playwright는 `e2e/` 아래 브라우저 테스트만 본다.

Java/Spring 관점으로 보면 unit test source set과 integration test source set을 분리한 것과 비슷합니다. 서로 다른 러너가 같은 테스트 파일을 건드리지 않도록 경계를 명확히 만든 것입니다.

## 셀렉터 안정화

E2E를 실제로 돌려보면, "보이기만 하면 된다"는 수준의 텍스트 셀렉터는 금방 흔들립니다. 이번 구현에서도 `AeroPress` 텍스트가 상세 화면에서 여러 번 등장해 Playwright strict mode와 충돌했고, 브루 스텝 textarea도 위치 기반 선택자는 취약했습니다.

그래서 최종 테스트는 다음 원칙으로 정리했습니다.

- 가능한 한 `getByRole`, `getByLabel`을 우선 사용
- 중복 가능성이 있는 버튼은 `exact: true`로 명시
- 브루 스텝 입력은 `aria-label="Brew step 1"`처럼 접근성 라벨을 추가해 안정적으로 선택
- 상세 화면 검증도 모호한 `AeroPress` 대신 `AeroPress Go`, `AeroPress Clear`처럼 구체적인 값으로 확인

이것은 테스트 편의용 해킹이라기보다, UI를 더 명확하게 설명 가능한 상태로 만드는 작업에 가깝습니다. 접근성 속성과 E2E 안정성이 같은 방향으로 맞물린 사례입니다.

## 실행 방법

한 번만 브라우저 바이너리를 설치합니다.

```bash
cd frontend
npm run test:e2e:install
```

이후 E2E는 아래 명령으로 실행합니다.

```bash
cd frontend
npm run test:e2e
```

## 이번 단계의 의미

이번 E2E 자동화는 모든 분기를 다루는 광범위한 회귀 테스트가 아닙니다. 대신 "서비스가 최소한 끝까지 동작한다"는 기준선을 세웠습니다.

앞으로 Phase 3 이후 자동완성, 인증, 멀티유저 제약이 추가되면 이 happy-path를 기반으로 시나리오를 점진적으로 확장할 수 있습니다.
